# AI Assistant 课程录制大纲与讲解脚本

## 1. 课程定位

这套课适合下面几类学员：

- 想做一个完整 RAG 项目的人
- 会一点 Java 或前端，但没真正做过 AI 项目的人
- 想把“能跑 Demo”提升到“能交付系统”的人

课程目标不是只教一个聊天框，而是让学员真正理解：

- 一个完整 RAG 产品由哪些模块组成
- 从文档入库到问答返回的完整链路
- 本地如何搭建
- 线上如何部署
- 出问题如何排查

## 2. 建议总时长

建议总时长：`8 到 12 小时`

如果你想做成比较完整的系列课，可以拆成：

- 入门篇
- 项目实战篇
- 部署上线篇
- 排障优化篇

## 3. 课程总目录

### 第一章：项目总览与效果展示

目标：

- 先让学员知道这个项目最终长什么样
- 先建立“全景认知”

建议演示：

- 打开用户端页面
- 打开管理端页面
- 演示上传文档
- 演示聊天问答
- 演示统计页和调用记录

建议讲点：

- 这不是一个单纯聊天框
- 这是一个完整企业级 RAG 系统
- 前台、后台、模型服务、向量库全部都有

### 第二章：基础概念扫盲

目标：

- 让小白先听懂术语

这一章一定要讲：

- 大模型是什么
- Prompt 是什么
- Embedding 是什么
- 向量库是什么
- Qdrant 是什么
- RAG 是什么
- Rerank 是什么
- SSE 是什么

讲法建议：

- 不要上来就讲学术定义
- 先用生活化语言解释
- 再映射到代码和系统模块

### 第三章：项目架构与模块划分

目标：

- 让学生知道代码应该去哪里看

建议讲解目录：

- `backend`
- `common`
- `consumer`
- `manager`
- `frontend/ai-assistant-consumer`
- `frontend/ai-assistant-manager`
- `deploy/server`

建议重点：

- 为什么现在用统一后端
- 为什么保留 `manager` 和 `consumer` 的模块边界
- 为什么部署时只启动一个 `backend.jar`

### 第四章：开发环境准备

目标：

- 让学生具备本地运行能力

建议讲这些：

- JDK 17
- Maven
- Node.js 18
- MySQL
- Redis
- Qdrant
- `.env.local` 配置

建议强调：

- 为什么 Java 版本必须是 17
- 为什么模型服务和数据库服务必须先准备好

### 第五章：本地一键启动与基础验证

目标：

- 让学员第一次把项目跑起来

建议操作：

```bash
./start-local.sh
```

这一章要讲清楚：

- 脚本做了什么
- 为什么是 9000、8001、9001 这三个端口
- 启动成功后该看哪些地址

建议顺序：

1. 先看后端健康检查
2. 再看管理端
3. 再看用户端

### 第六章：登录、权限和鉴权机制

目标：

- 让学生知道系统不是“谁都能调接口”

建议讲点：

- 验证码逻辑
- 登录逻辑
- JWT 生成
- `AuthorizationInterceptor`
- `UserContext`

重点文件：

- [UserController.java](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/consumer/src/main/java/com/osh/ai/assistant/consumer/controller/UserController.java)
- [UserServiceImpl.java](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/consumer/src/main/java/com/osh/ai/assistant/consumer/service/impl/UserServiceImpl.java)
- [AuthorizationInterceptor.java](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/common/src/main/java/com/osh/ai/assistant/common/interceptor/AuthorizationInterceptor.java)

建议断点：

- 登录接口
- 生成 token 的地方
- 拦截器解析 token 的地方

### 第七章：应用、知识库、文件三层关系

目标：

- 让学员理解业务模型

一定要讲明白：

- 应用不是知识库
- 知识库不是文件
- 文件上传后会进入知识库
- 聊天实际上是针对应用发起的
- 应用可以绑定知识库

建议画图：

- 用户
- 应用
- 知识库
- 文件
- 切片
- 向量

### 第八章：文档上传、解析与入库链路

目标：

- 让学员知道“上传文件”背后到底做了什么

建议讲链路：

1. 前端上传文件
2. 后端存储到磁盘
3. 解析文本内容
4. 切片
5. 调用 embedding
6. 写入 Qdrant
7. 记录 MySQL 元数据

建议重点：

- 文件原始内容不等于向量数据
- 向量库里存的是切片后的向量

建议断点：

- 文件上传接口
- 存储服务
- 文档解析器
- 向量写入逻辑

### 第九章：问答链路完整讲解

目标：

- 这是整套课最核心的一章

必须讲清楚：

1. 用户问题进入后端
2. 先做 embedding
3. 去 Qdrant 召回
4. 做 rerank
5. 拼接 prompt
6. 调用大模型
7. 通过 SSE 返回前端
8. 记录调用日志

重点文件：

- [AiChatServiceImpl.java](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/consumer/src/main/java/com/osh/ai/assistant/consumer/service/impl/AiChatServiceImpl.java)
- [ChatServiceImpl.java](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/consumer/src/main/java/com/osh/ai/assistant/consumer/service/impl/ChatServiceImpl.java)

建议断点：

- 聊天入口
- RAG 召回
- rerank
- 最终模型调用

### 第十章：SSE 流式返回原理

目标：

- 解释为什么回答是一段一段吐出来的

这一章要讲：

- SSE 与普通 HTTP 的区别
- `EventSource` 是什么
- 为什么前端需要先建立 SSE 连接
- 为什么聊天接口本身不是流式响应，而是通过已有 SSE 通道回推

建议结合本次真实故障案例讲：

- 原生 `EventSource` 发不出 `Authorization` 请求头
- SSE 连接没建成功时，聊天接口会出现“无效会话”
- 这次的修复方案是“query token + 后端兼容解析 + 前端发送保护”

这是很适合录课的真实案例，因为学生会觉得不是纸上谈兵。

### 第十一章：前端页面结构与交互设计

目标：

- 让前端同学知道页面是怎么组织的

建议讲：

- 登录页
- 管理后台
- 用户工作台
- 聊天页面
- 统计页面

适合讲的内容：

- 路由结构
- Pinia 登录态
- axios 请求拦截
- 聊天页 SSE 状态管理

### 第十二章：权限边界与安全修复

目标：

- 让学生知道企业项目里最容易被忽略的不是功能，而是权限

这一章建议重点讲：

- 为什么“知道 id”不代表“有权限操作”
- 资源必须和当前用户绑定校验
- 应用、知识库、文件、会话都要做归属校验

适合讲的价值：

- 这部分特别适合拔高课程质量
- 学员会意识到真正企业项目不是只把功能做出来

### 第十三章：本地调试与断点流程

目标：

- 帮助学员真正学会看代码

建议断点顺序：

1. 登录断点
2. 文件上传断点
3. 聊天断点
4. 调用记录断点

建议讲法：

- 每一章只打 2 到 4 个关键断点
- 不要把整条链全打满
- 否则小白会乱

### 第十四章：生产部署总览

目标：

- 让学生知道不是“本地能跑 = 项目结束”

建议讲：

- 生产依赖有哪些
- Docker Compose 是怎么编排的
- Nginx 怎么做代理
- 为什么统一后端更适合部署

建议重点服务：

- MySQL
- Redis
- Qdrant
- Embedding API
- Rerank API
- Backend
- Nginx

### 第十五章：Docker Compose 实战部署

目标：

- 带着学员走一遍真正可落地的上线流程

建议操作顺序：

1. 讲服务器目录结构
2. 讲 `.env`
3. 讲 `docker-compose.yml`
4. 讲 Nginx 配置
5. 讲后端生产配置
6. 启动容器
7. 检查健康状态
8. 浏览器访问验证

当前线上地址可以直接作为案例：

- `http://149.88.92.159/ai-consumer/`
- `http://149.88.92.159/ai-manager/`

### 第十六章：GitHub Actions 自动部署

目标：

- 让课程从“手工部署”提升到“工程化交付”

一定要讲：

- 为什么要自动部署
- GitHub Actions 做了哪些步骤
- 构建产物怎么上传
- 服务器怎么执行 `docker compose up -d`

建议强调：

- 自动部署不是炫技
- 它是减少手工失误、提高交付效率的工程能力

### 第十七章：常见问题排查

目标：

- 让学员面对报错时不慌

建议专门列一章讲：

- Java 版本问题
- Maven 找不到
- MySQL 连接失败
- Redis 验证失败
- Qdrant 不通
- Embedding 服务没起
- Rerank 服务没起
- SSE 连接失败
- 聊天页面显示“无效会话”

这一章非常适合结合你这次真实修复案例讲。

### 第十八章：课程总结与能力收束

目标：

- 帮学员形成完整认知闭环

总结时要让学生明白：

- 自己学到的不是一个页面
- 而是一整套 AI 应用工程链路

## 4. 适合录课的章节顺序

建议按这个顺序录：

1. 先展示最终效果
2. 再讲核心概念
3. 再讲项目结构
4. 再做本地启动
5. 然后讲登录和权限
6. 再讲知识库和文件
7. 再讲聊天主链路
8. 再讲前端 SSE

10. 最后讲排障和优化

这个顺序最适合小白，因为他们会先看到成果，再愿意听原理。

## 7. 录课时的讲解风格建议

建议你这样讲：

- 少讲空话，多讲真实链路
- 少堆术语，多用“输入什么、输出什么、经过什么”
- 每讲完一个点，就回到系统页面验证一次
- 每讲完一个模块，就落到一两个关键文件

不要这样讲：

- 一上来就贴一大段配置
- 一上来就讲太多抽象定义
- 一上来就带学生看 500 行代码

## 8. 可以直接拿来讲的三句总括

你可以在课程开头直接这么说：

第一句：

“这门课不是带你写一个聊天框，而是带你做一套真正能上线的 RAG 问答系统。”

第二句：

“你会学到的不是某一个接口，而是从文档入库、向量检索、重排序、模型生成，到前后端联调和服务器部署的完整链路。”

第三句：

“课程结束后，你至少应该具备三种能力：本地独立跑通项目、看懂核心问答链路、把项目部署到服务器并排查常见问题。”

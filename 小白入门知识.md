# OSH RAG 问答助手小白入门知识

这份文档是给第一次接触这个课程、第一次接触 RAG、第一次看这个项目的人准备的。

目标不是一下子把你变成专家，而是先让你知道三件事：

1. 这门课到底在做什么。
2. 这个项目里每个模块大概负责什么。
3. 你应该按什么顺序学，才不会一上来就被代码和部署劝退。

---

## 1. 先用一句人话讲清楚：这门课在做什么

这门课要带你做的是一个“带知识库的 AI 问答系统”。

你可以把它理解成：

- 普通大模型，像一个很聪明但没看过你公司资料的员工
- RAG 系统，像一个会先查资料、再回答问题的员工

所以这门课不是单纯教你“调用 AI 接口”，而是教你做一整套能真正落地的系统：

1. 上传文档
2. 解析文档内容
3. 把文档切成小段
4. 把文字转成向量
5. 存进向量数据库
6. 用户提问时先检索相关资料
7. 再让大模型根据资料生成答案

这就是 RAG。

RAG 的英文全称是：

`Retrieval Augmented Generation`

中文一般叫：

`检索增强生成`

你不用死记这个词，先记住一句话就够了：

`先找资料，再让 AI 回答。`

---

## 2. 为什么不能只调用大模型

很多新手最容易有一个误区：

“我直接接一个大模型接口，不就能做智能问答了吗？”

问题是，大模型虽然聪明，但它默认不知道你的私有资料，比如：

- 你的课程内容
- 你的企业制度
- 你的产品手册
- 你的售后 SOP
- 你的内部培训文档

如果没有把这些资料接进系统，大模型就容易：

- 答非所问
- 编造内容
- 回答不稳定

所以你需要给 AI 配一个“它自己的资料库”。

这也是这门课的核心价值：

不是让 AI 瞎猜，而是让 AI 带着依据回答。

---

## 3. 什么是 RAG，用故事理解

假设你开了一家卖课程的小店。

每天都有学员问你：

- 课程怎么开通
- 软件怎么安装
- 作业怎么提交
- 退款规则是什么

如果全靠你手工回复，你会很累。

于是你做了一个 AI 客服。

但这个 AI 一开始没看过你的资料，所以它答不准。

你就想到一个办法：

1. 先把课程文档、常见问题、规则说明都整理好
2. 有人提问时，先从资料里找到最相关的几段
3. 再把这几段资料交给 AI
4. AI 根据这些资料组织语言回答

这整套流程，就是 RAG。

所以你可以把 RAG 理解成：

`AI + 资料库 + 检索能力`

## 4. 这门课里会碰到的几个关键词

### 4.1 Embedding 是什么

Embedding 可以理解成：

`把一句话变成一串数字特征`

这样系统就能计算：

- 哪两段话意思更接近
- 用户的问题和哪段知识最像

你不用先关心数学细节，只要知道它的作用是：

`让计算机能比较“语义相似度”。`

### 4.2 向量是什么

向量就是 Embedding 之后得到的数字数组。

你可以把它粗略理解成：

`文字的数字指纹`

### 4.3 向量数据库是什么

普通数据库适合存：

- 用户名
- 密码
- 订单
- 文件记录

但不擅长高效查找“哪段文字和这句话最像”。

所以 RAG 项目通常会用专门的向量数据库。

这个项目里用的是：

`Qdrant`

它的职责很简单：

`存向量，并快速找出最相似的内容。`

### 4.4 Rerank 是什么

第一次检索出来的结果，不一定是最准确的。

这时系统会再做一次“重新排序”。

这一步就叫：

`Rerank`

它的作用是：

`把看起来相关的结果，再精排一遍，把最像答案的内容放前面。`

### 4.5 Context 是什么

Context 就是“上下文”。

在这个项目里，你可以把它理解成：

`最终喂给大模型参考的那几段资料。`

### 4.6 Prompt 是什么

Prompt 就是你给大模型的指令。

比如：

- 只根据资料回答
- 不知道就明确说不知道
- 语气要简洁

这些要求都会写进 Prompt。

## 5. 这个项目到底是不是完整 RAG

是，而且比较完整。

因为它不是只有一个聊天接口，而是已经把 RAG 的关键环节都串起来了：

1. 文档上传
2. 文档解析
3. 文本切块
4. 向量化
5. 向量库存储
6. 检索召回
7. 重排序
8. 上下文增强
9. 大模型回答
10. 流式输出
11. 会话记录
12. 后台管理

这意味着你学完之后，不只是懂一个概念，而是能看到一个真实可运行的项目长什么样。

## 6. 这个项目的目录，应该怎么理解

项目根目录下最重要的是这几个部分：

```text
osh-qa-assistant
├── backend
├── common
├── consumer
├── manager
├── frontend
│   ├── ai-assistant-consumer
│   └── ai-assistant-manager
├── deploy
├── scripts
└── 文档
```

你可以先这样记：

### 6.1 `backend`

统一后端启动入口。

它像“总开关”。

现在不是每个业务模块分别起一个服务，而是由一个统一后端把多个模块一起带起来。

### 6.2 `common`

公共模块。

这里放的是很多模块都会复用的能力，比如：

- 公共配置
- 工具类
- 异常处理
- 文件存储
- Redis 相关能力

### 6.3 `consumer`

用户侧业务。

可以理解成“普通用户使用的那一面”。

这里通常会涉及：

- 用户登录
- 知识库使用
- 文档管理
- AI 问答

### 6.4 `manager`

管理后台业务。

可以理解成“管理员使用的那一面”。

这里通常会涉及：

- 管理员登录
- 用户管理
- 应用管理
- 知识库总览
- 调用记录查看

### 6.5 `frontend/ai-assistant-consumer`

用户端前端。

技术栈是：

- Vue 3
- Vite
- Element Plus

### 6.6 `frontend/ai-assistant-manager`

管理端前端。

技术栈也是：

- Vue 3
- Vite
- Element Plus

### 6.7 `deploy`

部署相关目录。

主要是给服务器部署用的，比如 Docker Compose 配置。

### 6.8 `scripts`

脚本目录。

通常放一些辅助命令，比如本地验证、冒烟测试之类。

### 6.9 `文档`

课程文档和项目说明资料。

如果你是小白，这个目录非常重要，因为很多知识点和演示资料已经提前整理好了。

## 7. 这个项目里，几个核心组件分别干什么

### 7.1 MySQL 干什么

MySQL 主要存“结构化业务数据”，比如：

- 用户
- 管理员
- 应用
- 知识库记录
- 文件记录
- 聊天会话
- 聊天消息
- 调用日志

一句话记忆：

`MySQL 负责存业务数据。`

### 7.2 Redis 干什么

Redis 一般负责：

- 验证码
- 缓存
- 一些短期数据

一句话记忆：

`Redis 负责存临时、快速读写的数据。`

### 7.3 Qdrant 干什么

Qdrant 是向量数据库。

它最核心的任务是：

- 保存文档切片对应的向量
- 用户提问时做相似度检索

一句话记忆：

`Qdrant 负责帮你“找最像的知识”。`

### 7.4 大模型服务干什么

这个项目里，大模型相关能力通常分成三类：

1. Chat 模型
2. Embedding 模型
3. Rerank 模型

也就是说，这不是“一个模型做所有事”，而是不同模型各司其职。

一句话记忆：

`聊天、向量化、精排，往往不是同一个模型。`

## 8. 小白最该先建立的流程图思维

你先不要急着看细节代码，先把整个流程背下来。

### 8.1 文档入库流程

当你上传一个文档时，系统大概会做这些事：

1. 接收文件
2. 读取内容
3. 切成很多小段
4. 把每一段转成向量
5. 写进 Qdrant
6. 把文件和知识库关系记到 MySQL

这条线可以叫：

`入库线`

### 8.2 用户提问流程

当用户问一个问题时，系统大概会做这些事：

1. 接收问题
2. 把问题转成向量
3. 去 Qdrant 检索相似内容
4. 对检索结果做 rerank
5. 组装 Prompt 和上下文
6. 调用聊天模型生成答案
7. 把答案流式返回给前端

这条线可以叫：

`问答线`

如果你把“入库线”和“问答线”搞明白了，这个项目就已经懂了一大半。

`我知道一个可上线的 RAG 系统大致怎么搭起来。`

## 10. 小白第一次上手，最容易卡住的地方

### 10.1 把“会调用大模型”误以为“会做 RAG”

这是两个层级。

- 调大模型接口，只是会聊天
- 做 RAG，才是让聊天基于你的资料

### 10.2 一上来就想看懂所有源码

不现实。

正确顺序应该是：

1. 先懂业务目标
2. 再懂整体流程
3. 再认模块目录
4. 再跑项目
5. 最后再断点跟代码

### 10.3 一开始就死磕部署

如果你连项目是干什么的都没搞明白，直接看 Docker Compose，很容易越看越乱。

部署是后面的事，不是入门第一步。

### 10.4 对名词太焦虑

第一次看到这些词很正常：

- Embedding
- Vector
- Retrieval
- Rerank
- Context
- Prompt
- SSE

不用要求自己一次全懂。

你先知道它们分别在流程里“负责哪一步”，就已经足够开始学习了。

## 11. 小白应该按什么顺序学这门课

建议你按下面顺序来，不容易乱：

### 第一步：先看懂这份文档

目标：

- 知道课程是做什么的
- 知道 RAG 为什么存在
- 知道项目的大体结构

### 第二步：看已有的入门概念文档

推荐先看：

- `文档/0-入门概念解释.md`

这份更偏“故事化解释”，适合把陌生词先混个脸熟。

### 第三步：看完整教程或主手册

推荐继续看：

- `文档/无敌RAG项目完整实战教程.md`
- `课程讲解与部署实战手册.md`

这两份更像正式课程主线。

### 第四步：先跑通，再研究源码

先把项目跑起来，你才知道前端页面、接口、知识库、问答链路到底长什么样。

本地常见入口一般是：

- 后端：`http://127.0.0.1:9000/`
- 管理端：`http://127.0.0.1:8001/`
- 用户端：`http://127.0.0.1:9001/`

### 第五步：再做断点调试

你已经看过界面、点过功能、知道请求怎么走，再去断点才有意义。

否则你只会在代码里迷路。

这已经不是“只会调接口”的水平了，而是具备了看懂真实 AI 应用项目的基础。

## 13. 最后给小白的一句总结

你现在学的，不是“怎么让 AI 随便回一句话”。

`怎么把你自己的知识、文档、业务规则，接到 AI 身上，做成一个真正可用的问答系统。`

如果你先记住这句话，后面再看代码、看部署、看数据库、看向量库，就不会偏。

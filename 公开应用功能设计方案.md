# 公开应用功能设计方案

## 1. 背景与目标

当前项目已经具备应用、知识库、登录态聊天调试和开放 API 能力。现有开放 API 为 `POST /consumer/api/chat`，调用方需要传入 `appKey + appId + chatId`，更适合第三方系统后端或受控前端集成。

本次要设计的“公开应用”能力，目标是让应用拥有一个可访问的公开入口，其他人可以直接打开链接使用应用。公开入口支持两种访问方式：

- 公开访问：任何拿到链接的人都可以直接访问。
- 密码访问：访客需要输入访问密码后才能进入应用。

同时，平台管理员需要具备基础管理能力，例如查看公开状态、配置域名、统一巡检公开应用等。

核心原则：

- 不把用户的 `appKey` 暴露到公开页面。
- 不复用需要登录态的私有聊天接口作为公开入口。
- 公开能力独立建模，尽量不影响现有工作台、开放 API 和 RAG 聊天链路。

## 2. 当前代码现状

### 2.1 应用模型

当前应用实体位于：

- `common/src/main/java/com/osh/ai/assistant/common/bean/entity/AppDO.java`

现有 `app` 表字段主要包括：

- `id`
- `user_id`
- `app_name`
- `app_desc`
- `icon_path`
- `out_lib_enable`
- `lib_id`
- `created_time`
- `modified_time`

目前没有公开状态、访问密码、公开链接标识、域名等字段。

### 2.2 登录态聊天

当前工作台聊天入口主要位于：

- `consumer/src/main/java/com/osh/ai/assistant/consumer/controller/ChatController.java`
- `consumer/src/main/java/com/osh/ai/assistant/consumer/service/impl/ChatServiceImpl.java`
- `frontend/ai-assistant-web/src/views/personal/chat/Chat.vue`
- `frontend/ai-assistant-web/src/views/personal/chat/ChatList.vue`

这套接口依赖用户登录 token，通过 `UserContext` 校验用户是否拥有应用和会话。

### 2.3 开放 API

当前开放 API 位于：

- `consumer/src/main/java/com/osh/ai/assistant/consumer/controller/ApiController.java`
- `consumer/src/main/java/com/osh/ai/assistant/consumer/service/impl/ApiServiceImpl.java`
- `开放API对接手册.md`

接口特点：

- 免登录。
- 使用 `appKey + appId + chatId` 做身份与归属校验。
- 返回 SSE 流式结果。
- 适合系统对接，不适合直接暴露给普通访客页面。

### 2.4 权限白名单

后端白名单配置位于：

- `consumer/src/main/resources/application.yml`
- `backend/src/main/resources/application.yml`

前端路由白名单位于：

- `frontend/ai-assistant-web/src/permission/permission.ts`

公开应用页面和公开应用接口需要同时加入前端与后端白名单。

## 3. 建议方案

建议新增一套“应用发布配置”能力，而不是直接在 `app` 表上堆字段。

原因：

- 发布配置是应用的外部访问策略，和应用基础信息有独立生命周期。
- 后续如果扩展访问统计、域名绑定、过期时间、访问限流、嵌入代码等能力，独立表更稳。
- 可以保持 `app` 主表简单，降低对已有应用管理、开放 API、聊天调试的影响。

### 3.1 数据模型

新增表建议命名为 `app_publish_config`。

建议字段：

```sql
CREATE TABLE `app_publish_config`
(
  `id`             bigint       NOT NULL AUTO_INCREMENT COMMENT '主键',
  `app_id`         bigint       NOT NULL COMMENT 'app表主键',
  `enabled`        tinyint      NOT NULL DEFAULT 0 COMMENT '是否启用公开访问,1:是,0:否',
  `slug`           varchar(100) NOT NULL COMMENT '公开访问标识',
  `access_type`    varchar(20)  NOT NULL DEFAULT 'PUBLIC' COMMENT '访问方式:PUBLIC/PASSWORD',
  `password_hash`  varchar(200) DEFAULT NULL COMMENT '访问密码哈希',
  `custom_domain`  varchar(200) DEFAULT NULL COMMENT '自定义访问域名',
  `created_time`   datetime     NOT NULL DEFAULT current_timestamp() COMMENT '创建时间',
  `modified_time`  datetime     NOT NULL DEFAULT current_timestamp() COMMENT '修改时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_app_id` (`app_id`),
  UNIQUE KEY `uk_slug` (`slug`),
  UNIQUE KEY `uk_custom_domain` (`custom_domain`)
) ENGINE=InnoDB COMMENT '应用公开发布配置';
```

说明：

- `slug` 用于生成公开链接，例如 `/#/public/app/customer-service-demo`。
- `access_type = PUBLIC` 表示公开访问。
- `access_type = PASSWORD` 表示密码访问。
- `password_hash` 只存 BCrypt 哈希，不存明文。
- `custom_domain` 可为空，管理员后续配置域名时使用。

### 3.2 后端接口设计

登录用户侧接口建议放在 `consumer/app` 下，继续要求登录。

建议接口：

- `GET /consumer/app/publish/query?appId=xxx`
- `POST /consumer/app/publish/save`
- `POST /consumer/app/publish/disable`

公开访问侧接口建议新增独立 controller，例如 `PublicAppController`。

建议接口：

- `GET /consumer/public/app/detail?slug=xxx`
- `POST /consumer/public/app/verifyPassword`
- `POST /consumer/public/app/chat`

公开聊天接口建议由后端根据 `slug` 找到应用和所属用户，校验公开状态与访问凭证后，再复用 `AiChatService`。

关键点：

- 公开接口不接收 `appKey`。
- 公开接口不返回 `appKey`。
- 公开接口只返回公开页面需要的信息，例如应用名、描述、图标、是否需要密码。
- 访客会话 ID 由前端生成临时 ID 或由后端签发，避免所有访客共用一个会话上下文。

### 3.3 前端页面设计

工作台应用管理页增加发布设置入口。

涉及页面：

- `frontend/ai-assistant-web/src/views/personal/app/AppManage.vue`
- `frontend/ai-assistant-web/src/views/personal/app/AddApp.vue`
- `frontend/ai-assistant-web/src/views/personal/app/UpdateApp.vue`

建议新增组件：

- `frontend/ai-assistant-web/src/views/personal/app/PublishApp.vue`

工作台展示能力：

- 显示公开状态。
- 显示公开链接。
- 支持复制公开链接。
- 支持开启/关闭公开访问。
- 支持选择公开访问或密码访问。
- 支持设置或重置访问密码。
- 支持设置公开链接标识 `slug`。

公开访问页建议新增：

- `frontend/ai-assistant-web/src/views/public/PublicAppChat.vue`
- `frontend/ai-assistant-web/src/api/publicAppApi.ts`

公开页面能力：

- 未登录可访问。
- 根据公开配置展示应用名称、描述、图标。
- 密码访问模式下先展示密码输入。
- 验证通过后进入聊天页。
- 聊天结果使用流式输出。

### 3.4 管理员能力

管理员侧当前应用页主要是全量查看：

- `frontend/ai-assistant-web/src/views/admin/app/AppManage.vue`
- `manager/src/main/java/com/osh/ai/assistant/manager/controller/AppController.java`
- `manager/src/main/java/com/osh/ai/assistant/manager/service/impl/AppServiceImpl.java`
- `manager/src/main/resources/mybatis/mapper/AppMapper.xml`

建议第一期管理员能力保持克制：

- 查看应用是否公开。
- 查看公开链接标识。
- 查看是否配置自定义域名。
- 按公开状态筛选应用。

如果要支持管理员配置域名，建议新增独立接口：

- `POST /manager/app/publish/domain/save`

域名配置校验：

- 域名必须唯一。
- 域名只能绑定到一个公开应用。
- 域名为空表示不使用自定义域名。
- 更新域名前应校验应用存在。

## 5. 哪些地方不碰

为降低风险，第一期不建议改这些核心链路：

- 不改 RAG 检索、重排、提示词组装逻辑。
- 不改 `AiChatServiceImpl` 的核心问答流程，只通过新服务复用它。
- 不改现有登录态聊天接口行为。
- 不改现有 `/consumer/api/chat` 开放 API 协议。
- 不改上传、知识库入库、Qdrant、模型配置。
- 不改用户登录、注册、验证码逻辑。
- 不改现有私有会话历史查询逻辑。
- 不碰当前工作区已有未提交改动：`frontend/ai-assistant-web/src/views/personal/PersonalHome.vue`。

## 7. 最容易出问题的点

### 7.1 appKey 泄露

现有开放 API 依赖 `appKey`。公开页面如果直接复用开放 API，就很容易把 `appKey` 暴露到浏览器。正确做法是公开页面只传 `slug`、访客会话和用户输入，由后端根据发布配置找到应用与用户。

### 7.2 白名单遗漏

公开应用需要前端路由白名单和后端接口白名单同时配置。任何一边漏掉，都会出现未登录访问被拦截、跳登录页或返回鉴权失败的问题。

### 7.3 多访客上下文串线

公开链接会被多人同时访问。如果所有访客共用同一个 `chatId`，会导致上下文互相污染。公开访问必须给每个访客生成独立会话标识。

### 7.4 密码安全

访问密码不能明文保存。建议使用 BCrypt 哈希保存，并且公开详情接口不能返回 `password_hash`。

### 7.5 域名唯一性

自定义域名必须全局唯一。如果缺少唯一索引或保存前校验，可能导致一个域名对应多个应用。

### 7.6 SSE 生命周期

公开聊天如果使用 SSE，需要处理连接中断、重复发送、超时和浏览器刷新。后端应保证异常时能释放连接资源，前端应能展示失败状态并允许重试。

### 7.7 删除和关闭应用后的行为

应用删除、知识库解绑或关闭公开访问后，公开链接应立即失效。否则访客可能继续访问已不应公开的应用。

## 8. 建议实施顺序

第一阶段：基础公开访问

1. 新增发布配置表和实体。
2. 新增用户侧发布配置接口。
3. 新增公开详情接口。
4. 新增公开聊天接口。
5. 前端新增公开路由和公开聊天页。
6. 工作台应用列表增加公开设置弹窗。

第二阶段：密码访问

1. 增加访问类型配置。
2. 增加密码保存和校验。
3. 公开页增加密码输入状态。
4. 补充密码模式验证。

第三阶段：管理员能力

1. 管理员列表展示公开状态。
2. 管理员配置或查看自定义域名。
3. 增加公开状态筛选。

第四阶段：增强能力

1. 公开访问统计。
2. 访问限流。
3. 嵌入代码。
4. 公开链接二维码。
5. 公开应用过期时间。

## 9. 第一版取舍建议

第一版建议优先交付：

- 用户可开启/关闭公开访问。
- 用户可配置公开访问或密码访问。
- 用户可复制公开链接。
- 访客可打开公开链接并聊天。
- 管理员可查看公开状态。

第一版暂缓：

- 自定义域名自动解析检测。
- 公开访问统计看板。
- 嵌入式 iframe SDK。
- 复杂限流策略。
- 公开应用模板装修。

这样可以先把核心闭环打通，再逐步加运营和治理能力。

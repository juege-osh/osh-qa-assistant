## 2. 当前项目真实结构

- `backend`

## 3. 为什么之前会启动失败

你之前遇到的是：

- `UnsupportedClassVersionError`
- `class file version 61.0`
- `Java Runtime only recognizes class file versions up to 52.0`

这表示：

- 当前后端是按 `JDK 17` 编译的
- 运行环境如果还是 `Java 8`，就无法启动

对应关系：

- `61` = Java 17
- `52` = Java 8

所以当前方案统一要求：

- 本地运行后端必须用 `JDK 17`
- 服务器运行容器也必须使用 `Java 17` 运行时

## 12. 常见问题排查

### 12.1 后端起不来

先看日志：

```bash
docker logs -f ai-assistant-backend
```

重点排查：

- MySQL 是否可连通
- Redis 是否可连通
- Qdrant 是否可连通
- `backend-prod.yml` 是否挂载成功

### 12.2 本地报 Java 版本错误

先执行：

```bash
export JAVA_HOME=$(/usr/libexec/java_home -v 17)
java -version
```

确保显示的是 `17`

### 12.3 页面能开但接口报错

优先检查：

- 前端 `.env.development` 或生产代理是否指向统一后端
- Nginx 是否正确代理 `/manager/` 和 `/consumer/`
- 浏览器控制台是否有 `404`、`401`、`500`

### 12.4 问答或知识库异常

优先检查：

- DashScope API Key 是否有效
- Qdrant `6334` 端口是否可达
- `assistant_knowledge_lib` collection 是否存在
- 上传文件是否成功写入 `upload` 目录

### 12.5 验证码总是失败

优先检查：

- Redis 是否正常
- 应用是否能写入验证码键
- 登录和注册时的 `captchaId` 是否和当前验证码匹配

## 14. 最终结论

当前这套项目已经不是原来的“双后端双 Jar 独立运行”结构，而是：

- 一个统一 Spring Boot 后端：`backend.jar`
- 两个独立前端：manager + consumer
- 一个统一部署入口
- 保留原有 `/manager/**` 和 `/consumer/**` 接口体系

后续无论是本地启动、服务器部署、接口验证还是升级发布，都应该以这份文档为准，不再使用旧的 `manager.jar`、`consumer.jar`、`manager-prod.yml`、`consumer-prod.yml` 部署方式。

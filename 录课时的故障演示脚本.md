# AI Assistant 录课时的故障演示脚本

## 1. 这份文档怎么用

这份文档不是让你真的把线上环境搞坏，而是教你在录课时怎么“安全地演示故障”。

原则只有一个：

- 优先演示可控、可恢复、对主环境影响小的问题

不建议你录课时现场去破坏生产数据库或随便改服务器容器。

## 2. 最适合录课演示的 6 类故障

### 2.1 验证码过期

### 2.2 token 缺失或无效

### 2.3 SSE 对话异常

### 2.4 端口占用导致本地启动失败

### 2.5 基础服务未启动导致后端报错

### 2.6 知识库未绑定导致聊天条件不满足

## 3. 故障一：验证码过期

### 3.1 演示目标

让学员理解：

- 登录不是只传用户名密码
- 验证码是有时效的
- Redis 在这里扮演了缓存角色

### 3.2 演示步骤

1. 打开登录页
2. 先获取一次验证码
3. 不要立即提交
4. 等待验证码过期，或者刷新多次后故意用旧验证码提交
5. 提交登录

### 3.3 你可以这样讲

大家看，这里用户并不是输完用户名密码就能登录，还必须先带上 `captchaId` 和验证码结果。

如果验证码过期，后端在 Redis 里就查不到对应值了，所以会直接提示验证码不存在或已过期。

### 3.4 建议断点

- `UserServiceImpl.getCode`
- `UserServiceImpl.login`
- `validateCode`

### 3.5 预期现象

- 前端弹出验证码错误或过期提示
- 后端抛出 `验证码不存在或已过期`

## 4. 故障二：token 没带或 token 无效

### 4.1 演示目标

让学员理解：

- 登录成功不代表后续接口天然可用
- 受保护接口必须带 token
- 拦截器是整个鉴权体系的关键

### 4.2 演示方法一：浏览器开发者工具

1. 正常登录
2. 打开浏览器 Network
3. 找一个受保护接口
4. 观察 `Authorization` 请求头
5. 手工重放时去掉 token

### 4.3 演示方法二：Postman 或 curl

```bash
curl -X POST http://149.88.92.159/consumer/app/queryPage \
  -H 'Content-Type: application/json' \
  -d '{"pageNow":1,"pageSize":10}'
```

### 4.4 你可以这样讲

这里接口本身没有坏，坏的是调用方式。因为你没有带 token，所以拦截器在最前面就把请求拦掉了。

### 4.5 建议断点

- `AuthorizationInterceptor.preHandle`

### 4.6 预期现象

- 返回未授权或无效会话类提示

## 5. 故障三：SSE 对话异常

### 5.1 演示目标

这一段最适合讲真实线上问题。

要让学员理解：

- 普通 HTTP 接口和 SSE 接口不完全一样
- SSE 建连成功和能否正常发消息是两回事
- 前后端状态没对齐时，会出现“页面看似连着，实际上发不出去”的问题

### 5.2 推荐讲法

这里不建议你再把线上环境改坏。

最好的讲法是：

1. 直接讲“之前真实出现过什么问题”
2. 展示修复前后的代码差异
3. 再现场演示修复后的正常效果

### 5.3 真实问题点

- 缺少 `Authorization` 请求头时，SSE 场景没有取 URL 上的 `token`
- 前端连上 SSE 后，没有明确收到“连接已建立”的事件
- 页面卡在 `SSE 重连中`，导致用户误以为服务挂了

### 5.4 修复点

- 后端允许从查询参数读取 `token`
- 后端建连后立刻发 `connected`
- 前端收到 `connected` 后把状态切到已连接

### 5.5 对应代码

- [`AuthorizationInterceptor.java`](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/common/src/main/java/com/osh/ai/assistant/common/interceptor/AuthorizationInterceptor.java)
- [`ChatServiceImpl.java`](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/consumer/src/main/java/com/osh/ai/assistant/consumer/service/impl/ChatServiceImpl.java)
- [`Chat.vue`](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/frontend/ai-assistant-consumer/src/views/personal/chat/Chat.vue)

### 5.6 你可以这样讲

这个问题非常典型。很多项目普通接口都通，但一到流式接口就出问题，不是模型错了，而是连接管理和鉴权细节没处理好。

## 6. 故障四：本地端口占用

### 6.1 演示目标

让学员知道：

- 启动失败不一定是代码错
- 很多时候只是端口被占了

### 6.2 演示步骤

1. 本地先启动一份旧服务
2. 再执行 `./start-local.sh`
3. 观察报错

### 6.3 你可以这样讲

如果你看到 `9000`、`8001`、`9001` 端口冲突，不要先怀疑代码。先把已有进程停掉，再重新启动。

### 6.4 恢复方式

```bash
./stop-local.sh
```

必要时查看端口：

```bash
lsof -i :9000
lsof -i :8001
lsof -i :9001
```

## 7. 故障五：依赖服务没起导致后端报错

### 7.1 演示目标

让学员知道：

- Spring Boot 能启动，不代表业务一定完整
- MySQL、Redis、Qdrant、Embedding、Rerank 任何一个挂掉，都可能导致核心功能不可用

### 7.2 演示步骤

建议本地环境演示，不建议线上演示。

你可以在本地关闭 Redis 或 Qdrant，再启动后端。

### 7.3 你可以这样讲

真正项目里，应用本身只是中间层。数据库、缓存、向量库、模型服务共同组成了一条链路。链路上任何一个点挂掉，业务都会受影响。

### 7.4 建议检查顺序

1. MySQL
2. Redis
3. Qdrant
4. Embedding 服务
5. Rerank 服务
6. 后端配置文件

## 8. 故障六：知识库未绑定导致无法聊天

### 8.1 演示目标

让学员理解：

- 聊天不是只要创建会话就行
- 应用和知识库之间必须有绑定关系

### 8.2 演示步骤

1. 新建一个应用
2. 不绑定知识库
3. 直接去聊天
4. 观察前端提示或接口返回

### 8.3 你可以这样讲

这里的错误不是接口坏了，而是业务条件不满足。系统需要先知道这个应用应该查哪个知识库，否则就没法做召回。

## 9. 最适合录课展示的“问题到修复”案例

如果你想把课程讲得更像真实实战，我建议重点用下面这一个案例：

- 用户端聊天报“无效会话”
- 页面卡在 `SSE 重连中`
- 通过拦截器和前端 SSE 状态管理定位问题
- 修复 token 读取方式和 `connected` 事件
- 重新部署后验证恢复

这个案例的优点是：

- 足够真实
- 前后端都有内容
- 能讲鉴权、SSE、状态管理、部署和回归验证

## 10. 录课时的安全边界

建议做：

- 演示本地故障
- 演示修复前后的代码
- 演示修复后的线上结果

不建议做：

- 现场停线上 MySQL
- 现场改线上 Redis 密码
- 现场删除生产表字段
- 现场破坏线上容器

## 11. 每次故障演示后的收尾话术

每演示完一个故障，建议你都用类似的话术收尾：

这个问题本质上不是功能不会写，而是你有没有形成排障思路。真正线上项目里，最值钱的不是把 happy path 讲通，而是出了问题之后你知道先看哪里、怎么缩小范围、怎么验证修复是否生效。

## 2. 项目总览

### 2.1 项目目标

这个项目是一个典型的企业级 RAG 问答系统，目标是把“文档上传、切片、向量化、检索、重排序、问答生成、调用记录、后台管理”整合成一套完整可运行产品。

它同时包含：

- 管理后台
- 用户工作台
- 统一后端
- 向量数据库
- Embedding 模型服务
- Rerank 模型服务

### 2.2 模块划分

后端模块：

- `backend`
  - 统一启动入口
  - 把 `manager`、`consumer`、`common` 聚合到一个 Spring Boot 进程里
- `manager`
  - 管理后台接口
  - 管理员登录、应用管理、用户管理、知识库总览、调用记录等
- `consumer`
  - 用户端接口
  - 用户登录、应用管理、知识库管理、文档管理、聊天、统计等
- `common`
  - 共享配置、公共响应体、公共异常处理、Redis/MyBatis 配置、文件存储服务等

前端模块：

- `frontend/ai-assistant-manager`
  - Vue 3 + Vite + Element Plus
  - 面向管理员
- `frontend/ai-assistant-consumer`
  - Vue 3 + Vite + Element Plus
  - 面向普通用户

部署模块：

- `deploy/server/docker-compose.yml`
  - 统一生产编排入口
- `deploy/server/config/backend-prod-149.88.92.159.yml`
  - 当前生产环境后端覆盖配置
- `.github/workflows/deploy.yml`
  - GitHub Actions 自动构建与发布流程

## 3. 为什么现在采用“统一后端”

### 3.1 旧模式的问题

旧模式通常会把 `manager` 和 `consumer` 作为两个独立 Spring Boot 服务运行。这样会带来几个问题：

- 部署复杂
- 配置重复
- 端口更多
- 网关转发更碎
- 课程讲解成本更高

### 3.2 新模式的优势

现在统一由 [`backend/src/main/java/com/osh/ai/assistant/backend/BackendApp.java`](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/backend/src/main/java/com/osh/ai/assistant/backend/BackendApp.java) 启动：

- 一个 jar 就能承载全部后端功能
- 接口前缀保持不变
  - `/manager/**`
  - `/consumer/**`
- 更适合录课、部署、回归测试

核心思路：

1. `BackendApp` 扫描 `common`、`manager`、`consumer`
2. 排除 `ManagerApp`、`ConsumerApp` 这类独立启动入口
3. 用统一的 Spring 容器承载所有控制器、Service、Mapper

## 5. RAG 完整调用链

### 5.1 文档入库链路

用户上传文档后，后端大致会走这条链：

1. 上传文件到磁盘
2. 解析文件内容
3. 进行切片
4. 调用 embedding 服务生成向量
5. 把向量写入 Qdrant
6. 把文件、知识库、切片关系记录到 MySQL

关键文件：

- [`common/src/main/java/com/osh/ai/assistant/common/service/impl/StorageServiceImpl.java`](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/common/src/main/java/com/osh/ai/assistant/common/service/impl/StorageServiceImpl.java)
- [`consumer/src/main/java/com/osh/ai/assistant/consumer/elt/reader/TikaDocReader.java`](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/consumer/src/main/java/com/osh/ai/assistant/consumer/elt/reader/TikaDocReader.java)

本次修复的重点之一：

- 文件路径拼接从字符串拼接改为 `Path.of(...).normalize()`
- 文件预览统一复用 Tika 读取，避免 pdf/doc/docx 乱码

### 5.2 问答链路

一次聊天请求大致会走下面的步骤：

1. 用户发起问题
2. 根据 `appId` 找到关联知识库
3. 调用 embedding 服务把问题向量化
4. 到 Qdrant 做向量召回
5. 如果开启 rerank，再对召回结果重排序
6. 拼装提示词
7. 调用 Chat 模型生成答案
8. 记录调用日志与耗时、token 消耗等信息

- [`consumer/src/main/java/com/osh/ai/assistant/consumer/service/impl/AiChatServiceImpl.java`](/Users/rengang/chuangye/osh-projects/osh-rag/osh-qa-assistant/consumer/src/main/java/com/osh/ai/assistant/consumer/service/impl/AiChatServiceImpl.java)

## 14. 本次线上故障复盘

### 14.1 现象

之前线上表现是：

- 前台页面能访问
- 后端容器不断重启
- `19000` 不通
- 日志报 `Type com.osh.ai.assistant.consumer.ConsumerApp not present`

### 14.2 根因

根因不是代码逻辑错，而是：

- 直接在线上对 Spring Boot fat jar 做了错误的增量 patch
- 破坏了 nested jar 结构
- Boot Loader 无法正确加载 `consumer` 模块

### 14.3 修复方式

本次修复采用了更稳的路径：

1. 本地重新构建当前源码产物
2. 只上传小型 hotfix 包
3. 在服务器用 `eclipse-temurin:17-jdk` 容器重建完整 fat jar
4. 再整体替换 `/opt/qa-assistant/backend.jar`
5. 重启容器并验证健康检查

### 14.4 经验总结

以后不要再做的事：

- 不要直接 `zip append` 修改 fat jar
- 不要用错误方式重写 `BOOT-INF/lib/*.jar`
- 不要在坏包上继续叠 patch

推荐做法：

- 要么重新本地构建完整包上传
- 要么上传模块 jar 后在服务器重新完整打包

## 16. 常见问题排查

### 16.1 端口被占用

现象：

- 本地启动时报 `9000`、`8001`、`9001` 已被占用

处理：

```bash
./stop-local.sh
```

### 16.2 启动时提示 Java 版本不对

现象：

- `UnsupportedClassVersionError`

原因：

- 用 Java 8 运行了 Java 17 编译的包

处理：

```bash
export JAVA_HOME=$(/usr/libexec/java_home -v 17)
export PATH="$JAVA_HOME/bin:$PATH"
```

### 16.3 后端健康检查不通

优先检查：

1. MySQL 是否可连
2. Redis 是否可连
3. Qdrant 是否可连
4. Embedding API 是否 healthy
5. Rerank API 是否 healthy
6. 后端日志里是配置问题还是 fat jar 结构问题

### 16.4 Rerank 不生效

优先检查：

1. `AI_ASSISTANT_RERANK_ENABLED`
2. `AI_ASSISTANT_RERANK_PROVIDER`
3. `AI_ASSISTANT_RERANK_BASE_URL`
4. `rerank-api` 容器是否 healthy

### 16.5 文件预览乱码

如果又出现：

- 不要回退到 `Files.readString`
- 应继续走 Tika 解析链路

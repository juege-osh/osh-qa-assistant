# AI Assistant 项目常见问答 FAQ

## 1. 这套项目到底是不是一个完整 RAG 项目

是。

它不是只有一个聊天框，而是包含：

- 用户端
- 管理端
- 统一后端
- 文件上传
- 文档解析
- 向量化
- Qdrant 检索
- Rerank
- 模型问答
- 调用记录
- 统计分析
- Docker Compose 部署

## 2. 为什么有 `backend`、`consumer`、`manager`、`common` 四个模块

因为它把“启动入口”和“业务边界”分开了。

- `backend` 负责统一启动
- `consumer` 负责用户端业务
- `manager` 负责管理端业务
- `common` 负责共享能力

这比把所有代码硬塞在一个模块里更清晰，也比完全拆成多个独立微服务更适合教学和当前体量。

## 3. 为什么最终只启动一个后端 jar

因为当前项目已经做成了统一后端。

这样做的好处：

- 部署简单
- 配置集中
- 课程讲解成本低
- 回归测试路径更清晰

## 4. 用户端和管理端有什么区别

用户端是业务使用视角。

管理端是平台治理视角。

用户端更关注：

- 应用
- 知识库
- 文件
- 聊天
- 自己的调用记录

管理端更关注：

- 全部用户
- 全部应用
- 全部知识库
- 全部调用记录

## 5. 为什么聊天不是直接调用大模型

因为这是 RAG 项目，不是裸聊项目。

聊天前要先：

- 找应用绑定的知识库
- 做向量召回
- 需要时做 rerank
- 把召回结果注入 prompt

最后才调用大模型生成回答。

## 6. Embedding、Qdrant、Rerank 各负责什么

Embedding：

- 把文本转成向量

Qdrant：

- 存向量
- 查相似片段

Rerank：

- 对召回结果做二次排序

## 7. 为什么文件上传后还要“入库”

因为上传只是保存原始文件。

真正可检索还需要：

- 解析文本
- 切片
- 向量化
- 写入 Qdrant

所以“文件存在了”和“知识库可用了”是两回事。

## 8. 为什么聊天用 SSE

因为 SSE 很适合服务端连续推送文本片段。

用户看到 AI 一点一点输出，就是靠 SSE。

## 9. 为什么之前会报“无效会话”

核心原因是 SSE 场景下前后端鉴权和连接状态没处理好。

修复点包括：

- 后端支持从 URL 查询参数取 `token`
- 建连成功后后端主动发 `connected`
- 前端收到 `connected` 后切换连接状态

## 10. 为什么本地能用，线上却报错

最常见的原因不是“玄学”，而是下面几类：

- 线上没部署到最新代码
- 线上配置和本地不一致
- 线上依赖服务没起来
- 前端资源没更新
- SSE 或鉴权在代理层被影响

## 11. 为什么要单独做 `appKey`

因为开放 API 不能直接复用用户登录密码。

`appKey` 是给外部系统识别调用方用的，属于更适合对接的应用级身份凭证。

## 13. 录课时最值得讲哪几个点

如果你只想讲最值钱的内容，建议优先讲：

1. 验证码 + 登录 + JWT
2. 知识库、文件、应用三层关系
3. 文件上传到向量入库
4. RAG 召回与重排
5. SSE 流式输出
6. Docker Compose 部署
7. 真实线上故障排查

## 14. 这套项目后面还能往哪扩展

可以继续扩到：

- 多租户
- 更细权限体系
- 调用计费
- 模型切换
- 审计日志
- 工作流编排
- 多知识库检索策略

## 15. 录课时最怕被问什么

最容易被问的通常是：

- 为什么要拆这么多模块
- 为什么不是微服务
- 为什么用 Qdrant
- 为什么上传后还要入库
- 为什么用 SSE 不用 WebSocket
- 为什么线上和本地行为不一样

这份 FAQ 基本就是给这些问题准备的。

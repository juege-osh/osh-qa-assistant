Retrieval-Augmented Generation，检索增强生成。

先从外部知识库（如文档、数据库）中检索相关信息，再基于检索结果结合模型已有能力生成回答。不是训练，如用于企业内部知识库的智能问答、传统应用ai化、智能客服。

### qdrant安装与介绍

* 启动服务

  ```shell
  # 拉取镜像
  docker pull qdrant/qdrant:v1.16
  mkdir /root/qdrant 
  # 6333为控制台访问端口
  # 6334为服务端口
  docker run -p 6333:6333 -p 6334:6334 --name qdrant6334 \
  -v /root/qdrant:/qdrant/storage \
  -d qdrant/qdrant:v1.16
  ```

* 访问控制台

  http://ip:6333/dashboard

### spring ai中的elt

* Extract:抽取,对应DocumentReader接口,实现如:
  * TextReader
  * MarkdownDocumentReader
  * PagePdfDocumentReader
  * JsonReader
  * JsoupDocumentReader
  * tika(通用读取器)
* Transform:转换文档为分块,对应DocumentTransformer接口,实现如:
  * TokenTextSplitter:分隔文档为多个块
  * KeywordMetadataEnricher:关键字提取
  * SummaryMetadataEnricher:摘要提取
* Load:加载,在spring  ai中指的是写入向量数据库，对应DocumentWriter接口,实现如:
  * VectorStore
  * SimpleVectorStore
  * QdrantVectorStore

#### 基于重排序模型进行排序

* 为什么需要重排序

  向量数据库给出的Top-K只是粗排,向量数据库在毫秒级内捞出候选集，但有以下问题：

  1. 向量相似度 ≠ 真实相关性：向量数据库只看向量距离，可能把“字面像、实则无关”的片段排在前面。
  2. 近似搜索本身有精度损失：为保速度，向量数据库默认用降维、聚类或量化，会打乱原始顺序。
  3. 上下文理解缺失：价格、时效、用户画像等特征无法放进纯向量距离计算。

  生产级 RAG 系统都会再走一步重排序(rerank)：用交叉编码器、BERT 排序模型等对几十到几百条候选做细粒度打分。

  如用户提问"**感冒了能喝咖啡吗?**":

  向量数据库查到了3个文档，可能是：

  1. 距离最近（0.82）
     “**感冒时喝咖啡因饮料可缓解头痛，但会增加脱水风险……**”
     → 确实在聊“感冒+咖啡”，相关。
  2. 距离次近（0.79）
     “**马拉松赛后常出现感冒样症状，选手习惯在终点拿免费咖啡喝。**”
     → 出现了“感冒”“咖啡”两个词，但讲的是“赛后仪式”，根本没回答“能不能喝”。
  3. 第三近（0.77）
     “**咖啡因分子结构如下：C8H10N4O2，呈白色结晶粉末。**”
     → 只有“咖啡因”这个词跟问题字面重叠，语义完全无关。

  粗排阶段，2 和 3 因为“字面像”被打了高分；如果没有重排序模型把它们排除掉，直接送进 LLM，回答就容易跑偏。

#### 后端

1. 导入
2. 执行“1.表结构.sql"和"2.初始化数据.sql"

1. 修改application.yml

   * 数据库地址、端口、用户名和密码
   * redis地址、端口
   * 上传目录
2. 后台启动ManagerApp后访问即可: http://localhost:8000/manager/

##### c端服务

1. 修改application.yml

   * 数据库地址、端口、用户名和密码
   * redis地址、端口
   * 上传目录
   * 向量数据库collection-name、host、port

# RAG MVP 功能清单

## 1. 文档定位

这份文档用于明确本项目在 MVP 阶段到底要先做什么、后做什么、当前不做什么。

它不是研发任务拆解文档，而是产品优先级文档。

如果后续开发过程中出现以下情况，应优先回到这份文档判断：

- 新需求很多，但不知道是否应该进入当前版本
- 团队开始把精力分散到泛应用平台能力
- 想做的功能很多，但核心问答体验还没有稳定
- 管理能力、风控能力、装饰性能力开始挤占核心开发资源

## 2. 当前 MVP 目标

当前 MVP 目标不是做一个“大而全的平台”，而是做出一个：

- 以内部知识问答和任务指导为主场景
- 以高准确度为核心竞争力
- 能让最终用户稳定拿到可信结果
- 能让运营人持续优化知识效果

的一版基本可用产品。

一句话定义：

当前 MVP 的目标，是验证“高准确度 RAG 产品是否真正可用”，不是验证“平台是否足够丰富”。

## 3. 当前 MVP 成功标准

如果要判断 MVP 是否成立，至少要满足下面几条：

### 3.1 面向最终用户

- 用户能顺利提出问题
- 系统能返回与问题相关的回答
- 回答大多数情况下有知识依据
- 回答内容基本易懂，能支撑用户继续行动
- 没命中知识库时，不会直接抛出生硬错误

### 3.2 面向运营人

- 运营人能完成知识内容接入
- 运营人能完成基本应用配置
- 运营人能观察问答效果并做基础优化
- 运营人能区分正式版本与实验版本

### 3.3 面向交付

- 系统具备最小权限隔离
- 系统具备最小日志留痕
- 系统具备最小故障告警能力
- 系统具备基本发布和使用闭环

## 4. MVP 必须做（Must Have）

下面这些能力属于 MVP 阶段必须做的内容。

### 4.1 使用端核心能力

- 用户可以进入应用并发起提问
- 系统可以返回问答结果
- 问答结果尽量围绕知识库内容生成
- 支持任务指导类场景的结构化回答
- 回答失败时提供体面反馈

### 4.2 知识增强核心能力

- 知识内容接入
- 基础文档解析
- 文本切分
- 向量化入库
- 基础检索
- 基础重排或等价的效果优化能力
- 回答与知识内容关联

### 4.3 运营侧最小闭环

- 创建和维护知识内容
- 创建和维护应用
- 基础 Prompt / 模型开关配置
- 查看基础问答结果
- 对效果进行最小验证

### 4.4 体验保障能力

- 问答失败时不直接报错
- 未命中知识时给出明确说明
- 输出尽量结构化、可理解
- 对最终用户隐藏复杂 RAG 术语
- 能展示基础依据或参考来源

### 4.5 基础工程能力

- 登录与权限隔离
- 基础访问控制
- 日志留痕
- 基础故障告警

## 5. MVP 应该做（Should Have）

这些能力重要，但在资源有限时可以排在 Must Have 之后。

### 5.1 版本与实验能力

- 数据库层面对版本能力预留结构
- 支持实验版本与发布版本的概念区分
- 支持基础效果对比记录
- 支持不同切分策略的对比验证

### 5.2 运营可视化能力

- 让运营人看懂基础效果指标
- 指标解释足够通俗
- 用提示说明替代纯专业术语

### 5.3 体验优化能力

- 回答展示更清晰
- 任务指导回答支持更强结构化输出
- 对“依据充分/一般/未命中”做更友好的状态表达

## 6. MVP 可以延后（Could Have）

这些能力有价值，但不是当前 MVP 成立的必要条件。

### 6.1 泛应用扩展能力

- 陪伴型应用
- 人格型应用
- 风格化聊天应用
- 非知识库驱动的通用聊天玩法

### 6.2 完整版本管理能力

- 完整版本管理 UI
- 多维版本回滚面板
- 复杂实验编排系统

### 6.3 高级评测能力

- 更复杂的自动评测体系
- 多场景专项评测看板
- 更完整的实验统计分析

## 7. MVP 当前不做（Won't Have for Now）

下面这些内容在当前阶段明确不作为 MVP 交付目标。

### 7.1 非核心主卖点

- 把平台包装成泛 AI 应用平台
- 把“能搭很多种应用”作为当前版本主叙事
- 把陪伴、火星文、风格化玩法作为当前版本重点

### 7.2 完整平台治理体系

- 完整违规对话审核体系
- 完整风控等级体系
- 重型平台管理员能力
- 大而全的公开应用治理体系

### 7.3 装饰性优先能力

- 过早做复杂指标大盘
- 过早做复杂管理后台层级
- 过早做不直接影响问答质量的外围能力

## 8. 当前 MVP 的体验验收重点

MVP 是否成立，当前主要看下面几个体验点：

### 8.1 回答是否命中问题

- 是否答到了用户真正想问的问题
- 是否避免答非所问

### 8.2 回答是否可信

- 是否有知识依据
- 是否尽量减少无依据生成

### 8.3 回答是否易懂

- 是否结构清晰
- 是否适合执行
- 是否不需要用户自己二次翻译

### 8.4 失败是否体面

- 未命中知识时是否有明确提示
- 系统异常时是否避免粗暴报错
- 用户是否知道下一步可以怎么做

## 9. 面向运营人的指标建议

当前阶段建议运营侧围绕下面指标理解产品效果：

- `HitRate@5`
- `Answer Relevance`
- `Groundedness / Faithfulness`
- `Completeness`
- `Fluency / Structure`

但在产品表达上，不应只显示英文术语，建议同步转译为：

- 有没有检索到正确知识
- 回答有没有真正答题
- 回答有没有依据
- 回答有没有说全
- 回答是不是容易读懂

## 10. 当前 MVP 的开发约束

如果一个功能同时满足下面任意一种情况，应降低优先级：

- 不能直接提升最终用户问答体验
- 不能帮助运营人提高知识效果
- 只是让平台显得更“完整”
- 实现成本高，但当前无法验证对主场景的价值

如果一个功能同时满足下面两条，应优先进入 MVP：

- 能明显提升内部知识问答或任务指导效果
- 能明显降低最终用户获取答案的成本

## 11. 当前 MVP 的一句话范围

当前 MVP 要做的，不是“一个什么都能做的平台”。

当前 MVP 要做的是：

- 一个让最终用户在内部知识问答和任务指导场景下，真正愿意使用、能够信任、基本拿得到结果的高准确度 RAG 产品。

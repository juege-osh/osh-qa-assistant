# RAG 切分策略效果对比方案

## 1. 文档定位

这份文档用于定义本项目后续如何比较不同切分策略的效果，并逐步演进到半自动评测和智能推荐。

当前目标不是立刻做一个复杂的“智能切分平台”，而是先建立一套：

- 可比较
- 可复现
- 可留痕
- 可迭代

的实验机制。

这份文档主要服务：

- 知识库负责人 / 应用运营人
- RAG 效果优化
- 版本对比
- 后续智能判断能力建设

## 2. 为什么要单独做这件事

切分策略并不是一个纯技术参数问题，它会直接影响：

- 正确知识是否能被召回
- 回答是否能围绕正确上下文生成
- 是否出现断章取义、拼接错误、信息残缺

因此，切分策略不是“以后有空再看”的小优化，而是后续效果提升的重要抓手。

但它不能脱离评测闭环单独存在。

如果没有固定问题集、统一指标和记录方式，那么“哪种切分更好”往往只是主观判断。

## 3. 当前阶段的目标边界

### 3.1 当前阶段要做

- 定义切分实验的最小单位
- 定义切分对比时必须固定的变量
- 定义效果比较指标
- 定义实验记录模板
- 支持人工 + 半自动的结果判断

### 3.2 当前阶段不做

- 不直接做全自动智能选型
- 不直接做复杂实验编排系统
- 不直接做完整版本管理 UI
- 不直接做“模型自动决定一切”的黑盒系统

## 4. 一个切分实验到底比较什么

切分实验不是只改一个 `chunk size` 看看结果，而是比较“某一套切分方案”对最终问答结果的影响。

一个切分方案至少可能包含：

| 维度 | 示例 |
| --- | --- |
| 切分粒度 | `200 / 400 / 800` 字 |
| overlap | `0 / 50 / 100` |
| 标题继承方式 | 是否保留章节标题上下文 |
| 表格处理方式 | 表格按整块保留 / 转文本切分 |
| 列表处理方式 | 列表按项拆分 / 按段合并 |
| 特殊文档规则 | PDF、制度文档、FAQ 是否单独处理 |

因此，对比对象应该叫“切分策略版本”，而不是单一参数。

## 5. 对比实验的基本原则

为了确保对比结果可信，每次实验必须遵守下面原则：

### 5.1 一次只更换一个核心变量

如果本轮要比切分策略，就不要同时改：

- Prompt
- 模型
- 检索策略
- rerank 开关

否则无法判断效果变化是由什么引起的。

### 5.2 固定测试问题集

同一轮对比必须使用同一批测试问题。

建议至少固定：

- 标准知识问答
- 任务指导
- 模糊提问
- 未命中知识
- 失败场景

### 5.3 固定评测口径

不同实验必须使用同一套结论标准：

- 命中问题
- 可信
- 易懂
- 失败体面

以及同一套辅助指标：

- `HitRate@5`
- `Answer Relevance`
- `Groundedness / Faithfulness`
- `Completeness`
- `Fluency / Structure`

## 6. 如何比较不同切分方式

建议按下面顺序执行。

### 6.1 准备实验版本

先定义本轮要对比的版本。

示例：

| 版本 | 切分说明 |
| --- | --- |
| `Split-V1` | `chunk=400, overlap=50, 保留标题` |
| `Split-V2` | `chunk=800, overlap=100, 保留标题` |
| `Split-V3` | `chunk=400, overlap=50, 不保留标题` |

### 6.2 固定其他条件

本轮必须保持不变：

- 知识源一致
- 检索策略一致
- rerank 一致
- Prompt 一致
- 模型一致
- 应用配置一致

### 6.3 跑同一批测试问题

对每个版本都跑同一批问题。

推荐最少跑：

- 10 条基础题
- 5 条任务指导题
- 3 条模糊题
- 2 条未命中题

### 6.4 记录三层结果

每条问题至少记录：

1. 检索层
- Top-5 / Top-10 里有没有正确知识
- 正确知识排在第几位

2. 回答层
- 回答是否答题
- 回答是否有依据
- 回答是否完整

3. 体验层
- 用户是否容易理解
- 失败时是否知道下一步怎么做

### 6.5 汇总结论

最后不是简单说“感觉 V2 更好”，而是给出类似下面的结论：

- `Split-V2` 在任务指导问题上明显优于 `Split-V1`
- `Split-V2` 的 `HitRate@5` 更高
- `Split-V2` 回答更完整，但在模糊问题上仍有跑题
- 当前建议把 `Split-V2` 作为下一轮默认实验版本

## 7. 当前阶段建议的记录方式

当前最合适的方式不是直接做平台功能，而是先采用“实验记录文档 + 问题集文档”的组合。

建议至少使用两份文档：

1. `RAG_MVP测试问题集与效果记录`
2. `切分策略效果对比记录`

建议文件名：

- `YYYY-MM-DD_切分策略对比实验.md`

## 9. 能不能做成智能判断

可以，但建议分三阶段推进。

### 9.1 第一阶段：人工判断

当前最稳妥方式：

- 系统跑问题
- 人工查看召回结果和回答
- 人工打分

优点：

- 判断可靠
- 易落地
- 适合早期样本不足阶段

### 9.2 第二阶段：半自动判断

在人工基础上增加自动记录和规则判断。

可自动化的内容包括：

- 批量执行测试问题
- 保存召回结果
- 保存回答结果
- 自动统计 `HitRate@5`
- 自动标记是否命中预期知识

人工仍负责：

- 回答是否真正答题
- 回答是否易懂
- 回答是否体面

### 9.3 第三阶段：智能判断 / 推荐

当样本量和记录足够后，可以再引入：

- 规则评分
- LLM Judge 评分
- 历史实验结果聚合
- 针对文档类型的切分策略推荐

例如：

- 制度文档优先保留标题继承
- FAQ 文档适合更小粒度切分
- 操作手册适合“步骤块”切分

但在这之前，不建议直接把结果完全交给模型自动拍板。

## 10. 为什么这不是当前 P0，而是 P1

这件事已经是目标的一部分，但当前不放在 P0，有两个原因：

### 10.1 它依赖评测闭环先成立

如果下面这些还没立住：

- 固定问题集
- 统一验收口径
- 基础记录模板
- 基础通过/不通过判断

那么切分对比就没有稳定依据。

### 10.2 当前 MVP 更先要回答“系统能不能先稳定可用”

MVP 当前最优先的是：

- 回答命中问题
- 回答可信
- 回答易懂
- 失败体面

只有先把最小可用闭环立住，后续切分优化才有明确方向。

所以正确顺序应该是：

1. 先建立问题集
2. 先建立验收口径
3. 先能手工对比切分
4. 再做半自动对比
5. 最后做智能推荐

## 11. 后续建议进入开发的能力

当进入 P1 阶段后，建议按下面顺序开发：

1. 切分实验记录的数据结构
2. 实验版本与发布版本的关系定义
3. 批量跑问题集的执行入口
4. 结果记录与对比展示
5. 半自动评分
6. 智能推荐

## 12. 一句话结论

切分策略效果对比，是未来重要目标，而且必须做。

但当前阶段不应先做成复杂黑盒系统，而应先做成：

- 可比较
- 可记录
- 可复现
- 可逐步自动化

的实验能力。

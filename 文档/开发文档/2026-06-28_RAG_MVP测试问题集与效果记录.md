## 5. 验收口径

### 5.1 四项一级验收

每条回答都需要按下面四项打结论：

- 命中问题：是否真正答到了用户问题
- 可信：是否有知识依据，是否避免无依据生成
- 易懂：是否结构清晰，是否方便继续执行
- 失败体面：未命中或异常时，是否给出可理解、可接受的反馈

建议结论值统一为：

- `通过`
- `部分通过`
- `不通过`

### 5.2 运营侧辅助指标

为了方便非 RAG 背景的运营人理解，当前阶段建议同时记录下面几项：

| 指标术语 | 产品内通俗解释 | 建议记录方式 |
| --- | --- | --- |
| `HitRate@5` | 前 5 条里有没有检索到正确知识 | `有 / 没有 / 不确定` |
| `Answer Relevance` | 回答有没有真正答题 | `通过 / 部分通过 / 不通过` |
| `Groundedness / Faithfulness` | 回答有没有依据 | `通过 / 部分通过 / 不通过` |
| `Completeness` | 该说的有没有说全 | `通过 / 部分通过 / 不通过` |
| `Fluency / Structure` | 是否容易读懂并执行 | `通过 / 部分通过 / 不通过` |

  - 真实题集还不够能拉开差距

- 把切分预览证据一起带进跨版本对比脚本，避免后续只看到通过率，看不到切分层证据

当前这条链路已经变成：

- 发布某个实验版本为当前生效版本

这已经比“有入口能点”更接近真正的 MVP 运转闭环。

### 12.11 高风险外部问题失败反馈再收紧

在继续往真实闭环推进时，本轮又发现了一个更贴近可用性的细节：

- 对于 `RQ-05` 这类高风险外部问题，系统虽然已经不会直接强答
- 但在弱相关召回存在时，仍可能顺带给出泛化建议
- 例如引导用户去看某类外部机构或公共来源

这会让“当前知识库没有直接依据”的边界显得不够干净，也会影响“可信”和“失败体面”的稳定性。

本轮修正点：

- 收紧 `AiChatServiceImpl` 中高风险外部问题的兜底逻辑
- 当问题命中高风险关键词，且当前召回依据仍不足以直接支撑回答时：
  - 直接返回受控拒答文案
  - 不再继续让模型补充外部泛化建议
- 同时修正 `RagAcceptanceServiceImpl` 的自动评估口径：
  - 像这种“正确拒答”的场景，不再误判成普通调用失败
  - 会识别为：
    - `grounded=通过`
    - `graceful=通过`
    - `followUpCategory=''`

| 问题 | 结果 |
| --- | --- |
| `请直接告诉我美国联邦所得税 2026 年怎么报。` | 不再给出外部泛化建议，直接明确“当前知识库里没有直接依据”，并建议补充制度名称、流程来源或联系专业负责人 |
| `如果我要开始做训练测试任务，第一步应该先做什么？` | 正常回答不受影响，仍能给出直接结论和后续步骤 |

这说明当前又补齐了一层更真实的 P0 质量要求：

- 不是只要“拒答了”就算过
- 而是要保证：
  - 高风险外部问题不强答
  - 拒答边界干净
  - 验收评估口径也能正确理解这种受控拒答

### 12.12 补知识并复跑的工程化闭环

前面已经能自动识别：

- 是切分问题
- 还是提示词问题
- 还是知识内容本身不够清楚

但如果每次补知识都还要人工做下面这些动作，闭环仍然不够稳：

- 手工整理补知识文档
- 手工上传到验收知识库
- 手工删除上一版补丁文件
- 手工重建索引
- 手工切换生效实验版本
- 手工复跑真实问题集

所以本轮继续补了一条专门面向“知识修补”的脚本化闭环：

- 脚本：`scripts/rag-mvp-knowledge-repair-run.sh`

这条脚本会自动完成：

1. 生成针对当前问题的补知识文档
2. 删除同名旧补丁文件，避免重复叠加
3. 上传新补丁到 `RAG MVP 验收知识库`
4. 发布目标实验版本为当前生效版本
5. 以当前生效版本重建整个知识库索引
6. 复跑同一组真实问题集正式验收
7. 输出逐题结论与批次汇总

当前这份补知识文档重点针对：

- `RQ-01` 对知识库覆盖范围概括不够稳定

补的核心内容不是继续泛讲“系统能做什么”，而是把边界直接写死为三类：

1. 内部文档知识问答
2. 训练测试任务的起步与执行顺序
3. 知识没命中时的体面反馈

这样做的目的，是让语义切分版本在回答“这个知识库现在主要能回答哪些类型的问题”时，也更容易稳定命中正确原文，而不是泛化成“知识库系统操作和管理流程”。

这条脚本补上之后，后面每次发现：

- 某题是 `knowledge`
- 某题是 `chunking`
- 或两者混合

都可以先走“补知识 -> 重建 -> 复跑 -> 看正式结果”这条确定链路，而不是凭感觉改完再手工试。

### 12.13 补知识复跑真实结果

本轮没有停留在“脚本能跑”，而是继续把 laughing 本地运行态一起验证到位。

中间还定位出一个容易误判的问题：

- 一次并行执行里先重启了 laughing，再完成后端重新打包
- 导致 `.laughing-runtime/backend.jar` 和容器里的 `/app/backend.jar` 其实还是旧包
- 表面上看像是“代码没修对”，实际是运行包没有同步到最新版本

把这条部署顺序问题修正之后，再次执行：

- 重新打包 `backend.jar`
- 重新同步到 `.laughing-runtime`
- 重新启动 laughing 应用栈
- 再跑 `scripts/rag-mvp-knowledge-repair-run.sh`

最终真实批次：

| 批次 ID | 生效实验版本 | 切分策略 | 结果概览 |
| --- | --- | --- | --- |
| `2071738549524738050` | `脚本语义切分版` | `semantic` | `items=6, pass=6, followUp=0` |

这说明当前已经可以确认两件事：

1. `RQ-01` 这类“知识库覆盖范围”问题，确实可以通过补知识内容本身来修复
2. `RQ-05` 这类“高风险外部问题正确拒答”场景，自动验收口径也已经改到位，不再把正确拒答误判成命中失败

当前这一轮真正打通的是：

- 补知识文档生成
- 上传替换旧补丁
- 按当前生效版本重建索引
- 真实问题集正式验收复跑
- laughing 运行态结果验证

这意味着后面如果再暴露新的 `knowledge / chunking / prompt` 问题，我们已经有了一条可以重复执行、能直接产出正式验收批次的修复闭环。

### 12.14 当前 P0 审计结果

本轮在 laughing 环境真实执行结果：

- 跨切分版本对比：`PASS`
- 生效切分版本：`PASS`
- Prompt / 模型开关：`PASS`
- 告警 readiness：`DISABLED`

其中告警 readiness 的真实状态为：

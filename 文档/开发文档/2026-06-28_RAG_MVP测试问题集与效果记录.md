# RAG MVP 测试问题集与效果记录

## 1. 文档定位

这份文档用于承接当前 RAG MVP 的测试问题集、回答记录和效果判断。

目标不是做复杂评测系统，而是先建立一套可以重复执行、可以留痕、可以横向比较的最小闭环。

这份文档主要服务：

- MVP 自测
- 功能验收
- 版本对比
- 体验复盘

## 2. 使用说明

每次准备验收一个版本时，建议复制一份本文件，按当次版本填写。

## 5. 验收口径

### 5.1 四项一级验收

每条回答都需要按下面四项打结论：

- 命中问题：是否真正答到了用户问题
- 可信：是否有知识依据，是否避免无依据生成
- 易懂：是否结构清晰，是否方便继续执行
- 失败体面：未命中或异常时，是否给出可理解、可接受的反馈

建议结论值统一为：

- `通过`
- `部分通过`
- `不通过`

### 5.2 运营侧辅助指标

为了方便非 RAG 背景的运营人理解，当前阶段建议同时记录下面几项：

| 指标术语 | 产品内通俗解释 | 建议记录方式 |
| --- | --- | --- |
| `HitRate@5` | 前 5 条里有没有检索到正确知识 | `有 / 没有 / 不确定` |
| `Answer Relevance` | 回答有没有真正答题 | `通过 / 部分通过 / 不通过` |
| `Groundedness / Faithfulness` | 回答有没有依据 | `通过 / 部分通过 / 不通过` |
| `Completeness` | 该说的有没有说全 | `通过 / 部分通过 / 不通过` |
| `Fluency / Structure` | 是否容易读懂并执行 | `通过 / 部分通过 / 不通过` |

## 7. 建议默认问题池

下面这组问题可以作为第一版默认问题池，后续再按具体知识库补充。

### 7.1 标准知识问答

- 这个知识库当前主要覆盖哪些主题？
- 某个制度/流程的核心结论是什么？
- 某个常见业务问题应该参考哪份文档或哪类规则？

### 7.2 任务指导

- 我应该从哪里开始完成这个任务？
- 这个任务的准备项、输入项和关键注意事项分别是什么？
- 如果我要把这个流程走通，先后步骤应该是什么？

### 7.3 模糊提问

- 这个事情我该怎么弄？
- 我现在不知道从哪开始，你建议我先看什么？
- 这块一般要注意什么？

### 7.4 未命中知识

- 询问一个明确不在知识库范围内的问题
- 故意使用模糊、歧义、缺少关键词的问题

### 7.5 失败场景

- 检索结果为空时，系统是否明确说明未找到足够依据
- 链路异常时，系统是否给出体面提示而不是直接报错

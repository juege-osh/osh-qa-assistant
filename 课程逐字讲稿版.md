## 2. 开场白讲稿

大家好，这套课我们不做那种只能跑一个聊天框的小 Demo，我们直接做一个完整的企业级 RAG 问答系统。

这套系统里会包含前台用户工作台、后台管理端、统一后端、知识库、文件上传、向量检索、重排序、流式对话、调用记录、统计分析，以及最后的 Docker Compose 部署和自动化发布。

也就是说，你学完之后拿到的不是一个零碎功能点，而是一整套从开发到上线、从功能到排障都能讲清楚的完整项目。

这一套项目现在已经部署在线上，访问入口有两个。一个是用户端 `http://149.88.92.159/ai-consumer/`，一个是管理端 `http://149.88.92.159/ai-manager/`。

接下来我会先带你看效果，再讲架构，再讲本地启动，再讲核心源码链路，最后讲服务器部署、自动发布和线上排障。

## 3. 第一章讲稿：先看项目最终效果

现在我们先不急着看代码，先看项目跑起来之后到底长什么样。

我先打开管理端。管理端主要是给管理员用的，可以看用户、应用、知识库、上传文件和调用记录。它更偏平台视角。

然后我再打开用户端。用户端是业务使用方真正会操作的地方。用户可以注册登录、创建应用、创建知识库、上传资料、发起聊天、查看调用情况。

这里你要先建立一个概念，这个项目不是“用户上传文件然后直接丢给模型”，而是先把文件解析、切片、向量化、存进向量库，然后提问时再做召回和增强生成。

所以它本质上是一套 RAG 系统，而不是单纯的聊天系统。

## 4. 第二章讲稿：先把概念讲明白

在开始看代码之前，我们先把几个最重要的概念讲清楚。

第一个，大模型。大模型负责生成最终答案。我们这个项目现在接的是 OpenAI Compatible 协议，实际调用的是 `MiniMax-M2.7-highspeed`。

第二个，Embedding。你可以把 Embedding 理解成把一句话变成一串数学向量。因为数据库没法直接理解语义，所以要先转向量，再去做相似度检索。

第三个，向量库。我们这里用的是 Qdrant。Qdrant 不负责生成答案，它只负责存向量、查相似内容。

第四个，Rerank。你可以把它理解成二次排序。先粗召回一批片段，再由 Rerank 模型重新判断谁更相关，让真正有用的片段排到前面。

第五个，SSE。SSE 是服务端推送。我们在聊天页面看到 AI 一个字一个字往外吐，就是通过 SSE 来实现的。

如果这些词你以前只在文章里看过，没有在项目里真正串起来，那这套课正好把这些概念全部落到真实代码和真实接口上。

## 5. 第三章讲稿：项目架构总览

下面我们看项目结构。

后端这里有四个核心模块，分别是 `backend`、`consumer`、`manager`、`common`。

`backend` 是统一启动入口。也就是说最终真正启动的是一个 Spring Boot 进程，而不是 manager 一个进程、consumer 一个进程分开跑。

`consumer` 负责用户端接口，比如用户登录、应用管理、知识库管理、文件上传、聊天、统计。

`manager` 负责管理端接口，比如管理员登录、用户管理、应用总览、知识库总览、调用记录查看。

`common` 放的是共享能力，比如 JWT、拦截器、统一返回体、Redis、文件存储之类。

前端这边有两个 Vue 3 工程，一个是 `frontend/ai-assistant-consumer`，一个是 `frontend/ai-assistant-manager`。

部署上，现在也已经统一成了一套 Docker Compose，所以你本地讲完开发，再讲上线时逻辑是连贯的。

## 6. 第四章讲稿：本地环境准备

接下来我们讲本地环境。

这个项目后端要求 Java 17，Maven 3.8 以上。前端建议 Node 18。

基础依赖服务至少包括 MySQL、Redis、Qdrant。

除此之外，还有两个模型相关服务。一个是 Embedding 服务，一个是 Rerank 服务。当前部署方案里，Embedding 用的是 `BAAI/bge-small-zh-v1.5`，Rerank 用的是 `BAAI/bge-reranker-base`。

这里要特别强调一点，很多同学本地跑不起来不是代码有问题，而是基础依赖没准备好。比如数据库没启动、Redis 密码不对、Qdrant 没起、Embedding 服务没准备好，这些都会导致后端启动失败或者聊天不可用。

## 7. 第五章讲稿：本地一键启动

现在我们进入项目目录，执行 `./start-local.sh`。

这个脚本会自动帮我们做几件事：检查 Java 版本、加载 `.env.local`、检查端口、构建后端、启动后端、启动两个前端，并等待 3 个地址可访问。

启动之后，我们先不要急着操作页面，而是先验证后端健康检查。

打开 `http://127.0.0.1:9000/`，如果返回 `UP`，说明后端进程已经起来了。

然后再打开管理端 `http://127.0.0.1:8001/`，最后打开用户端 `http://127.0.0.1:9001/`。

这一段录课时建议你一定要强调顺序。先看后端，再看前端。因为很多排障问题本质上不是页面问题，而是后端没起来。

## 8. 第六章讲稿：登录、验证码、JWT 鉴权

下面我们讲登录链路。

不管是用户端还是管理端，登录前都要先获取验证码。验证码不是存在数据库里，而是短暂存在 Redis 里。

用户点击获取验证码之后，后端会生成一张图片，计算出答案，然后把答案放进 Redis，同时返回一个 `captchaId` 和图片 Base64。

用户提交登录时，会把用户名、密码、`captchaId`、验证码结果一起传给后端。

后端先根据 `captchaId` 去 Redis 校验验证码，校验通过后再查数据库、比对密码、生成 JWT token。

接下来前端访问受保护接口时，就会把 token 放进 `Authorization` 请求头里。

这里你可以在三个地方打断点。

第一个断点打在 `UserServiceImpl.getCode` 或管理端对应的 `getCode`。

第二个断点打在 `UserServiceImpl.login`。

第三个断点打在 `AuthorizationInterceptor.preHandle`。

这样一条登录链路，验证码、Redis、数据库、JWT、拦截器全能讲清楚。

## 9. 第七章讲稿：应用、知识库、文件这三层关系

这一章一定要讲明白，否则学员后面会一直混淆。

第一层是应用。应用是面向聊天场景的业务入口。

第二层是知识库。知识库存的是业务资料集合。

第三层是文件。文件属于知识库，文件上传后会被解析和切片。

所以关系不是“应用下面直接挂文件”，而是应用绑定知识库，知识库下面再挂文件。

提问时，用户其实是针对某个应用发起聊天。系统再根据应用关联到知识库，进而去召回对应知识片段。

你在这里最好画一张图，把“用户 -> 应用 -> 知识库 -> 文件 -> 切片 -> 向量”这个关系讲透。

## 10. 第八章讲稿：文件上传、解析、入库

下面我们来演示一个最关键的业务流程，就是文档入库。

用户先创建知识库，然后上传一个 PDF 或 markdown 文件。

很多初学者以为上传完成就等于知识库可用了，其实不是。上传只是把原始文件存起来，真正可检索还要经过解析、切片、向量化和写入 Qdrant。

这一段背后的核心步骤是：

第一，把文件保存到磁盘。

第二，把文件内容解析出来。

第三，按一定长度切成多个片段。

第四，调用 Embedding 模型生成向量。

第五，把向量写入 Qdrant。

第六，把文件、知识库、文档片段等元数据写进 MySQL。

这就是为什么我们说原始文件和向量数据不是一回事。

## 11. 第九章讲稿：聊天问答完整链路

接下来我们讲最核心的对话流程。

当用户在聊天页输入问题之后，系统会先根据 `appId` 找到绑定的知识库。

然后把问题先转成向量，到 Qdrant 里召回一批相似片段。

如果当前启用了 Rerank，就会把召回片段再过一遍重排序模型，筛出更相关的内容。

然后系统会把这些知识片段和用户问题一起拼进 Prompt，再调用大模型生成最终答案。

如果前端开启的是流式聊天，那么后端会通过 SSE 一段一段把内容推回来，前端就能边收边显示。

最后，系统还会把本次调用记入 `invoke_record` 和 `invoke_record_detail`，把聊天消息写入 `chat_message`。

这一段是整套课最有价值的部分，因为它把 RAG 真正跑起来了。

## 12. 第十章讲稿：SSE 为什么之前会出问题

这一章适合讲真实线上故障。

前面这个项目有一个很典型的问题，就是用户端聊天时会报“无效会话”，或者页面一直卡在 `SSE 重连中`，导致用户以为服务坏了。

这个问题不是模型有问题，也不是数据库有问题，而是 SSE 连接场景下前后端鉴权和连接状态处理没对齐。

后端这次修复做了三件事。

第一，SSE 接口在没有 `Authorization` 请求头时，允许从 URL 查询参数里取 `token`。

第二，SSE 连接建立后，后端立即返回一个 `connected` 事件。

第三，前端收到 `connected` 事件后，把连接状态改为已连接，不再一直卡在重连态。

这段内容特别适合录课，因为它非常真实，能让学员感受到项目不是只讲 happy path，而是会真正讲线上故障和修复思路。

## 13. 第十一章讲稿：断点调试该怎么看

如果你要带学员真正看懂项目，我建议至少打下面几个断点。

第一个，登录接口断点。看验证码、密码校验、token 生成。

第二个，文件上传断点。看原始文件保存路径。

第三个，文档解析断点。看提取出来的文本内容。

第四个，切片断点。看一个文件如何变成多个 document。

第五个，向量写入断点。看写进 Qdrant 的 payload 和向量。

第六个，召回断点。看召回出来了哪些文档片段。

第七个，Rerank 断点。看二次排序前后顺序有什么变化。

第八个，SSE 断点。看后端每次推送了什么，前端每次收到了什么。

第九个，鉴权拦截器断点。看 token 是怎么解析的。

如果这些断点都演示到位，学员对整套系统的理解会比只看页面强很多。

## 14. 第十二章讲稿：管理端和用户端的区别

很多同学第一次看这种项目，会把用户端和管理端混在一起。

这里要明确区分。

用户端关注的是业务使用。比如建应用、建知识库、上传资料、发起聊天、看自己调用情况。

管理端关注的是平台管理。比如看全体用户、看全量知识库、看所有调用记录、做平台级巡检。

你录课时最好左右各开一个窗口，一边是 manager，一边是 consumer。这样学员会立刻理解这是两个不同角色，不是同一个后台套两层菜单。

## 15. 第十三章讲稿：部署架构怎么讲

下面我们进入部署部分。

当前生产服务器是 `149.88.92.159`，SSH 端口是 `16328`，部署目录是 `/opt/qa-assistant`。

线上现在统一由 Docker Compose 编排，核心服务包括：

- `mysql8`

其中后端对外暴露的是 `19000`，Nginx 负责把用户端和管理端静态页面对外发布，并把接口反代给后端。

所以用户最终访问的是：

- `http://149.88.92.159/ai-consumer/`

## 16. 第十四章讲稿：自动化部署怎么讲

除了手工部署，这个项目还有 GitHub Actions 自动部署。

逻辑是这样的。

代码推到 `main` 或 `release/**` 分支之后，工作流会自动构建后端 jar、构建两个前端、通过 SSH 把产物同步到服务器，然后在服务器执行 `docker compose up -d --remove-orphans`。

最后还会做健康检查，验证后端 `19000` 是否正常返回。

这一段录课时你不一定要现场跑一次 GitHub Actions，但你至少要把工作流文件打开，让学员知道自动发布不是一句空话，而是仓库里已经写好了。

## 17. 第十五章讲稿：如何演示一次完整业务闭环

如果我现在带你完整走一遍项目，我建议按这个顺序演示。

先登录管理端，看平台视角。

再登录用户端，创建一个普通用户自己的知识库。

然后上传一个文档，执行入库。

接着创建一个应用，绑定知识库。

然后新建会话，提一个知识库内问题。

再提一个知识库外问题，演示严格模式和宽松模式区别。

最后打开统计页和调用记录页，给学员看系统不是只会回答问题，还会记录调用情况和耗时。

这一条链路跑通之后，学员就会真正理解“完整项目”和“零散功能”之间的区别。
